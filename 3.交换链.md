## render path(渲染流程)

RenderProcess 目前主要是在做“图形流水线状态定义”，不是在描述完整的数据来源。

pipeline layout（你代码里的 pipeline_layout_），它的作用是“定义 shader 能访问的外部资源接口”。

可以把它理解成：
它不是具体数据本身，而是一份“资源插槽协议”。

它主要约束两类东西：

Descriptor Set Layouts
告诉 Vulkan：shader 会用到哪些资源类型和绑定位，比如：
binding 0 是 uniform buffer
binding 1 是 combined image sampler
这些资源在哪个 shader stage 可见
Push Constant Ranges
定义一小段可快速更新的常量数据（几十到几百字节），以及它在 shader 里的可见阶段。

为什么需要它：

在创建 graphics pipeline 时，pipeline 必须绑定一个 layout。
运行时你绑定 descriptor set、push constants 时，必须和这个 layout 匹配。
不匹配就会触发 validation error，或者行为未定义。
和 render pass 的区别：

pipeline layout：管“资源接口”（UBO/纹理/push constant）。
render pass：管“附件与子通道流程”（颜色附件格式、load/store、layout 转换等）。

====================
对，完全正确。

pipeline layout 只定义“接口契约”，不负责真正传数据。它回答的是：

要传哪些类型的数据（UBO、Sampler、Push Constant）
每个数据在什么 set/binding 上
哪些 shader stage 可见
它不回答的是：

数据什么时候更新
数据从哪里来（CPU 内存、staging buffer、每帧计算等）
用什么命令提交绑定
真正“传递数据”发生在运行时命令里，比如：

vkUpdateDescriptorSets 或写 descriptor buffer
vkCmdBindDescriptorSets
vkCmdPushConstants
（如果是顶点数据）vkCmdBindVertexBuffers
所以可以记成一句话：
layout 是类型系统/函数签名，cmd 绑定和 update 才是实参传递
